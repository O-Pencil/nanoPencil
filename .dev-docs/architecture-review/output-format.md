# Output Format

Schemas for the three Phase 2 artifacts. The Arch Agent must produce all three; their formats are not negotiable (so that future runs can be compared).

---

## 1. Finding card

File: `.dev-docs/architecture-review/findings/F<NN>-<short-slug>.md`

`<NN>` is two-digit zero-padded sequence number. `<short-slug>` is lowercase, non-alnum → `-`, max 60 chars.

```markdown
# F<NN>: <Title — short noun phrase, ≤ 80 chars>

```yaml
finding_id: F<NN>
severity: load-bearing | structural | opinionated
lenses: [depth, seam, leverage, locality, DIP]   # at least one; multi-lens findings are stronger
files_primary:
  - <repo-relative path>
files_secondary:
  - <repo-relative path>
discovered_in_phase: 1
status: open                                       # open → selected | rejected (in Phase 3)
```

## Problem

One paragraph in plain English. What's structurally off here? What does it cost a maintainer / future feature work today?

- Concrete observation 1 with `file:line` evidence
- Concrete observation 2 with `file:line` evidence

Avoid vague language ("hard to understand", "messy"). Use the methodology vocabulary: shallow / missing seam / poor locality / depth mismatch / DIP drift.

## Deletion test

> "If module `<X>` were deleted, would complexity concentrate in callers, vanish, or stay roughly the same?"

**Result**: concentrates / vanishes / inconclusive.

Brief rationale: <one paragraph>.

If `vanishes` → strong signal the module is shallow scaffolding.
If `concentrates dramatically` → module is doing real work but might be over-loaded — recommend splitting, not deleting.
If `inconclusive` → either the test is wrong for this finding, or the finding is opinionated (downgrade severity).

## Proposed direction

**Not a full design.** Just the seam to introduce or the depth to add.

- New seam at: `<file:line>` — abstract over `<X>` so it can vary with `<Y>`.
- Or: merge `<file-a>` into `<file-b>` because the deletion test shows nothing was hidden.
- Or: extract a deep module covering `<concern>` from the current scattering across `<list>`.

Keep this section short. Phase 3 is where the design gets fleshed out.

## Benefits

In leverage / locality terms (methodology.md §1.4):

- **Leverage**: <who gains, what they gain>
- **Locality**: <what becomes editable in one place>

If neither is concretely articulable, downgrade severity to `opinionated`.

## Before / after sketch

```
BEFORE                                AFTER

<ASCII or Mermaid showing today's     <ASCII or Mermaid showing the
 coupling/depth shape>                 proposed shape>
```

For HTML report, embed Mermaid. For the markdown card, ASCII is fine.

## ADR / DIP conflict callouts (optional)

If this finding conflicts with an explicit assertion in any `CLAUDE.md` (P1/P2) or `.PENCIL.md`:

- Conflict: <P-level reference + quoted assertion>
- Resolution: revise the P-doc, accept the finding, or accept the conflict and document why.

## References

- mattpocock methodology: SKILL.md (depth/seam/leverage)
- DIP protocol: root `CLAUDE.md` §"DIP Dual-phase Isomorphic Documentation Protocol"
- Adjacent findings: F<MM>, F<KK> (if related)
```

### Finding card sizing

A good card is 200–600 lines of markdown. Below 200: probably under-developed; ask whether the finding is actually `opinionated`. Above 600: probably trying to solve the problem in the card; punt the design to Phase 3.

## 3. HTML overview

File: `$TMPDIR/architecture-review-<YYYYMMDDhhmm>.html` (or `/tmp/` on Linux, `%TEMP%` on Windows)

Self-contained. No external resources. Inline CSS, inline SVG, embedded Mermaid (rendered server-side at generation; or embedded source with mermaid.js loaded from CDN — but CDN load is acceptable since this file is read locally by the maintainer).

Sections:

1. **Header**: timestamp, total findings, severity breakdown.
2. **Refactor plan summary** (collapsed list, sortable by severity / lenses).
3. **One card per finding**, in priority order, each containing:
   - Title + severity badge (🟥/🟧/🟨)
   - Files involved (with relative paths)
   - Problem statement
   - Deletion test result
   - Proposed direction
   - Benefits
   - Before/after sketch (Mermaid where applicable)
   - Link back to the source markdown card
4. **ADR conflict section** if any.
5. **Adjacent observations** (things the Arch Agent noticed that aren't its territory — handed to the self-diagnosis side, see `handoff.md`).

The HTML is the maintainer's primary surface. Optimize for **fast scanning**: severity colors, anchor links between findings, "back to top" links. The HTML is **read-only** — never edited after generation. If a finding changes in Phase 3, regenerate the HTML.

```markdown

```yaml
finding_id: F<NN>

phase: 3
```

## Original finding (verbatim)

<copy the card>

## Maintainer rejection reason

<what the maintainer said, in their own words or your paraphrase, attributed>

## Architectural cost of rejection

What future work becomes slower / harder / impossible if this stays unaddressed?

- ...

## Re-open conditions

If <X> happens, this finding should be revisited. Examples:
- A new feature lands that depends on the seam this finding proposed.
- A future Arch Agent run independently surfaces the same friction.
```

## 5. What the output is NOT

- **Not a PR.** Findings are recommendations. Implementation lives in separate work.
- **Not auto-applied changes.** The Arch Agent never edits source code (`extensions/`, `core/`, `packages/`, etc.).
- **Not a benchmark report.** Performance is not in scope (methodology.md §4).
- **Not a security audit.** Security is not in scope.
- **Not exhaustive.** Better to ship 8 strong findings than 30 weak ones.
