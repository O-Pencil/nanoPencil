## Ordering rationale

1. **先治环（F03/F04），再拆 god（F01/F02），最后做体积/启动（F06/F07）**。原因：在循环依赖下做大文件拆分会触发雪崩式 type 重建与意外 import 链路，先把环消掉是低成本前置。
2. **F08 与代码改动并行**。它本身不改代码，但**例外白名单**机制是 F01–F07 不再回流的安全网；提前布点，每个 finding 落地时同步删除对应 exception。
3. **F07 排最后**，因为 dist 体积是"所有上述决策的下游产物" —— bundle 工具与 lazy 入口设计必须先有 F06，才能用 esbuild 干净分片。

## Phase 2.5 — Top-Level Structure & Extensibility Review

**评审目标升级**（来自 maintainer 要求）：从"在现有骨架内做减法"升级为"先问骨架本身对不对"。新增 3 个评审目标：项目结构合理性 / 功能合理性 / 扩展能力。

- **文档**：[`./top-level-structure-review.md`](./top-level-structure-review.md)

关键新发现：

- **packages/ 是"形式上的多包，实质上的单包"** —— 5 个包发布到 npm（其中 soul-core 404 未发布），但 **0 个外部消费者直接 import**；host `dependencies` 也不含 `@pencil-agent/*`；packages 全部通过 `bundle-deps.js` "倒灌" 进 host dist。`workspaces` 字段实质只承担内部代码组织。
- **3 处真背离 README 初衷**：
  1. README "Privacy First — no telemetry" vs 1.14.3 实际加入远程 telemetry
  2. README "Extensible — Plugin system" vs 现实只有 first-party + MCP（无 user-dir loader / 无第三方 theme / 无 extension-sdk 稳定包）
  3. README 三层架构（Cognitive / Tool / Interface）与代码四分目录（packages / core / modes / extensions）**完全正交，无映射**
- **`core/` 实际是"杂物间"** —— 业务核心（runtime/tools/mcp）+ 横切基础设施（i18n/telemetry/utils）+ Interface 边角（keybindings）混装

提出 3 个顶层候选方案（详见 top-level-structure-review §6）：

- **A 认领 monorepo**（主推）—— 承认 packages/ 是内部组织，引入 `src/` `platform/` `cognitive/` 与 README 三层对齐
- **B 认领 ecosystem packages** —— packages 升到 Pencil 顶层，真做生态级共享
- **C 保守** —— 不动骨架，只修 3 处真背离

**新决策点 Q10–Q14**：
