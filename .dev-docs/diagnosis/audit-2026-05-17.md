Today every analytics query on this table has to cast `text → int` on the fly, which silently swallows malformed rows.

## 3. Capture-chain integrity

### 3.1 `eval_runs` lifecycle is leaking

53 of 92 runs are stuck at `status=running`. Oldest is from **2026-04-26T17:13Z** — three weeks ago. The `insforge-sink` code (`§eval/insforge-sink.ts:268`) PATCHes the row on `run_end`. Per `sal/index.ts` the SAL extension has "emergency flush on beforeExit/SIGHUP/SIGTERM". Reality: that flush is missing ~58% of runs.

| Status | Count | Oldest start | Newest start |
|--------|-------|--------------|--------------|
| running | 53 | 2026-04-26 | 2026-05-11 |
| completed | 39 | 2026-04-23 | 2026-05-10 |
| failed | 0 | n/a | n/a |

→ Two follow-up questions: (a) why does the emergency flush miss; (b) should a janitor sweep mark stuck-running rows as `failed` after some TTL (the SAL code mentions `cleanup_stale_runs` is opt-in — check whether it ever runs).

### 3.2 `eval_tool_traces.intent = "unknown"` (65.7%) — classifier weak

The sink at `insforge-sink.ts:327` reads `taskSignals?.intent`. The producer that feeds `task_signals` is on the SAL side. Either the producer rarely sets `intent`, or its taxonomy doesn't cover most turns. Both are field-quality bugs.

### 3.3 No referential integrity checks

PostgREST shows zero foreign keys across these tables. `eval_turns.run_id`, `eval_tool_traces.run_id`, `eval_memory_recalls.run_id` all reference `eval_runs.run_id` informally but the database doesn't enforce it. With 53 stuck-running parent rows, we cannot tell from the DB whether a turn's parent run actually exists or just hasn't reported its end yet. Worth re-examining if we want analytics across runs.

### 3.4 `pencil_issue_events` types are stored as TEXT

Identical to §2.4 disease, on a smaller table. `occurrence_count`, `first_seen_at`, `last_seen_at`, `recorded_at` are all `text`. Every query in the SOP today does on-the-fly casts (`NULLIF(...,'')::int`) which masks bad rows.

## 6. Schema recommendations

These are **all REVIEW** per SOP §3.3 (telemetry write side). Filing as recommendations, not auto-fixes.

### 6.1 `pencil_issue_events`

- **Drop** `mode`, `thinking`, `tool_summary`, `user_note` — or wire the writers if intent was real
- **Type-fix** `occurrence_count text → int`, `first_seen_at / last_seen_at / recorded_at text → timestamptz`
- **Type-fix** `severity` → `text` with check constraint `IN ('debug','info','warning','error')`
- Add an index on `(fingerprint, created_at DESC)` — the SOP's daily query scans full table; will hurt as it grows

### 6.2 `eval_runs`

- **Drop** `task_file`, `config_snapshot`, `total_tokens_used`, `error_message` unless populated
- Add janitor: any `status='running'` older than 24h → `status='abandoned'` with note `auto-closed: no run_end received`
- Add foreign key `eval_turns.run_id → eval_runs.run_id ON DELETE CASCADE` once janitor runs

### 6.3 `eval_turns`

- **Drop** `assistant_response_length`, `tokens_used`, `tools_called` (the last is shadowed by `eval_tool_traces`)
- Or — if response-length + tokens are wanted — keep them and **wire the writer**; that's the more useful direction since both are basic per-turn cost signals

### 6.4 `eval_tool_traces` vs `eval_tool_calls`

- Two paths, both REVIEW:
  - **(P)** Promote `eval_tool_calls` to canonical (properly typed). Migrate the writer in `insforge-sink.ts:316–348` to target `eval_tool_calls`. Backfill the 382 historical traces. Drop `eval_tool_traces`.
  - **(Q)** Accept `eval_tool_traces` as canonical. Drop `eval_tool_calls`. Type-fix `eval_tool_traces` columns in place (TEXT → INT/BOOL/TIMESTAMP) — riskier because of the existing 382 rows.
- Recommendation: **(P)** — keep the well-typed schema, port the writer, accept the one-time backfill cost.

### 6.5 `eval_sal_anchors`

- **Drop** `source`, `unresolved_signals`, `weights_source` unless the SAL extension is going to start populating them
- Otherwise the schema misleads anyone reading it

## 7. What this means for the daily SOP (Phase 2)

Today's SOP (`docs/daily-issue-sop.md`) queries only `pencil_issue_events`. Three implications from this audit:

1. **The SOP's queries cast `occurrence_count` to int with `NULLIF(...,'')::int`.** That trick survives malformed rows silently. Worth a stricter assertion if the type-fix in §6.1 lands.
2. **Section §3 ("classification") doesn't mention the dead fields.** If `tool_summary` / `thinking` were live, the SOP could use them to triage faster. Right now those columns can't help.
3. **The SOP doesn't reach into `eval_*` at all.** A fingerprint like `mem-core.extract:fallback:work-llm-failure` would be much easier to triage if the SOP cross-referenced `eval_turns.user_prompt` and `eval_tool_traces.tool_sequence` for the same `session_id`/`run_id`. Suggested follow-up: add an §X "Cross-reference to eval data" step to the SOP.
