## Event vocabulary (reference)

| Event | Terminal? | Meaning |
|-------|-----------|---------|
| `ticket-opened` | no | BLOCK or REVIEW ticket filed at `runs/<date>/<slug>.md` |
| `ticket-closed` | **terminal** | the docs PR carrying this ticket was merged (or the maintainer manually closed it) |
| `auto-fix-pushed` | no | the agent pushed an `auto/issue-<...>` branch + opened the PR |
| `auto-fix-merged` | **terminal** | the auto-fix PR was merged to main |
| `auto-fix-rejected` | **terminal** | the auto-fix PR was closed without merging |
| `attempt-failed-demoted` | no | the agent tried AUTO-FIX, tsc/vitest failed, demoted to REVIEW; a follow-up `ticket-opened` event is logged the same day |
| `observed` | no | added to today's OBSERVE watchlist |
| `observed-aged-out` | **terminal** | fingerprint left the 7-day baseline without escalation |
| `skipped` | **terminal** | the day was SKIPPED (MCP outage, dirty tree, low resources); `Fingerprint = *` |

A fingerprint is **outstanding** when its latest event is non-terminal. The Outstanding section above is computed by walking the event log bottom-up and listing one entry per fingerprint whose latest event isn't in the terminal set.

## Maintenance notes

- The Diagnosis Agent appends events at end of each run and regenerates the Outstanding section before pushing.
- The Review Agent does **not** edit this file. It reads the Outstanding section to give context to its review comments when relevant, but never writes.
- Archive rotation (SOP §9.3) does **not** move this file. The ledger stays in `runs/`. If event-log volume becomes a problem (likely beyond several thousand rows), the maintainer manually splits the log into a yearly `LEDGER-<YYYY>.md` and recreates a fresh one — that step is not automated.
- The maintainer may add explanatory notes between rows in the event log if a particular event needs context, **but the table columns must stay parseable** by the agent.
