# NanoPencil `/loop` 命令开发计划

## 一、背景与目标

参考 Claude Code 的 `/loop` 命令（详见 `docs/Loop.md`），为 NanoPencil 实现同等能力的定时循环任务调度功能。用户可通过 `/loop` 命令在会话中安排重复性或单次的自动化提示词任务，实现 PR 监控、CI 修复、定时检查等自动化场景。

### 核心目标

- 用户可通过 `/loop <间隔> <提示词>` 安排定时执行的 agent 任务
- 支持灵活的时间间隔语法（秒、分、时、天）
- 任务为会话级作用域，关闭会话即销毁
- 提供任务列表查看、删除等管理能力
- 设定合理的安全限制（最大任务数、有效期、费用保护）

---

## 二、功能规格

### 2.1 `/loop` 命令语法

```
/loop <prompt>                     # 默认每 10 分钟执行一次
/loop 30m <prompt>                 # 前置时间令牌，每 30 分钟执行一次
/loop <prompt> every 2h            # 后置 every 从句，每 2 小时执行一次
/loop list                         # 列出所有调度任务
/loop delete <taskId>              # 删除指定任务
/loop clear                        # 清除所有任务
```

### 2.2 时间单位

| 单位 | 含义 | 示例     |
| ---- | ---- | -------- |
| `s`  | 秒   | `30s`    |
| `m`  | 分钟 | `10m`    |
| `h`  | 小时 | `2h`     |
| `d`  | 天   | `1d`     |

- 最小粒度：**1 分钟**（秒级单位向上取整到 60s）
- 默认间隔：**10 分钟**
- 不均匀间隔自动舍入到最接近的"整洁"值

### 2.3 任务生命周期

- **会话级作用域**：任务仅在当前 NanoPencil 进程中存活，关闭会话即销毁
- **硬性有效期**：所有重复性任务最长存活 **3 天**，到期后执行最后一次触发并自动删除
- **最大并发数**：单个会话最多 **50 个**调度任务
- **执行优先级**：任务在 agent 空闲时（两个用户 Turn 之间）以低优先级入队执行
- **无补录策略**：如果 agent 忙碌时任务到期，空闲后只触发一次，不补偿漏掉的次数

### 2.4 防抖动机制

- 为每个任务分配基于 taskId 的确定性偏移量（jitter）
- 重复性任务：最多延迟周期的 10%（上限 15 分钟）
- 避免多任务在整点同时触发导致 API 流量激增

## 三、架构设计

### 3.1 整体架构（已实现：内置扩展）

`/loop` 以**默认内置扩展**实现（`extensions/defaults/loop/`）：`pi.registerCommand("loop", …)` 注册斜杠命令；调度器在扩展进程内持有，在 **`session_shutdown`**（会话结束、`/reload` 等）时 **`dispose`**。触发执行使用 **`pi.sendUserMessage`**：空闲时直接发起 turn；Agent 流式处理中则 **`deliverAs: "followUp"`** 入队。

扩展 API 增补 **`pi.isIdle()`**，供定时回调在无 `ExtensionContext` 时判断排队策略。

```
extensions/defaults/loop/index.ts
  → loop-parser / loop-scheduler
  → pi.sendUserMessage（必要时 followUp）
  → AgentSession（既有）
```

### 3.2 模块划分

#### 实现文件（扩展目录）

| 文件路径 | 职责 |
| -------- | ---- |
| `extensions/defaults/loop/index.ts` | 扩展入口：`/loop` 命令、`session_shutdown` 清理 |
| `extensions/defaults/loop/loop-scheduler.ts` | 任务 CRUD、setTimeout 链式调度、jitter、到期删除 |
| `extensions/defaults/loop/loop-parser.ts` | 间隔与参数字符串解析 |
| `extensions/defaults/loop/loop-types.ts` | 类型定义 |

#### 核心配合（薄改动）

| 文件路径 | 修改内容 |
| -------- | -------- |
| `builtin-extensions.ts` | 注册 `extensions/defaults/loop` |
| `package.json`（`copy:extensions`） | 构建拷贝到 `dist/extensions/defaults/loop` |
| `core/extensions/types.ts` | `ExtensionAPI` / `ExtensionRuntime`：`isIdle()` |
| `core/extensions/runner.ts` / `loader.ts` | 绑定与占位 |

> 原计划中的 `core/slash-commands` 内置注册、`interactive-mode` 专页分支、`AgentSession` 内嵌 `_loopScheduler` **未采用**；对外语法与行为与本文第二节一致。

## 四、详细设计

### 4.1 类型定义（`extensions/defaults/loop/loop-types.ts`）

```typescript
export interface LoopTask {
  id: string;                    // 8位随机字符 ID
  prompt: string;                // 要执行的提示词
  intervalMs: number;            // 执行间隔（毫秒）
  createdAt: number;             // 创建时间戳
  expiresAt: number;             // 过期时间戳（创建后 3 天）
  lastExecutedAt: number | null; // 最后一次执行时间
  executionCount: number;        // 已执行次数
  jitterMs: number;              // 防抖动偏移量
  timerId: ReturnType<typeof setTimeout> | null; // 定时器句柄
}

export interface LoopSchedulerConfig {
  maxTasks: number;              // 最大任务数，默认 50
  maxLifetimeMs: number;         // 最大存活时间，默认 3 天
  defaultIntervalMs: number;     // 默认间隔，默认 10 分钟
  minIntervalMs: number;         // 最小间隔，默认 1 分钟
  maxJitterRatio: number;        // 最大 jitter 比例，默认 0.1
  maxJitterMs: number;           // jitter 上限，默认 15 分钟
}
```

### 4.2 命令解析器（`extensions/defaults/loop/loop-parser.ts`）

解析三种语法：

1. **管理命令**：`/loop list`、`/loop delete <id>`、`/loop clear`
2. **前置时间**：`/loop 30m check the build`
3. **后置 every**：`/loop check the build every 2h`
4. **纯提示词**：`/loop check the build`（使用默认间隔）

```typescript
export interface ParsedLoopCommand {
  type: "create" | "list" | "delete" | "clear";
  prompt?: string;
  intervalMs?: number;
  taskId?: string;       // delete 时使用
}

export function parseLoopCommand(input: string): ParsedLoopCommand;
export function parseInterval(token: string): number | null;
export function formatInterval(ms: number): string;
```

### 4.3 循环调度器（`extensions/defaults/loop/loop-scheduler.ts`）

```typescript
export class LoopScheduler {
  private tasks: Map<string, LoopTask> = new Map();
  private config: LoopSchedulerConfig;
  private executeCallback: (task: LoopTask) => Promise<void>;

  constructor(
    config: Partial<LoopSchedulerConfig>,
    executeCallback: (task: LoopTask) => Promise<void>,
  );

  // 创建新任务
  create(prompt: string, intervalMs?: number): LoopTask;

  // 删除指定任务
  delete(taskId: string): boolean;

  // 清除所有任务
  clear(): void;

  // 列出所有任务
  list(): LoopTask[];

  // 销毁调度器（会话结束时调用）
  dispose(): void;

  // 内部：安排下一次执行
  private scheduleNext(task: LoopTask): void;

  // 内部：执行任务
  private executeTick(task: LoopTask): Promise<void>;

  // 内部：检查任务是否过期
  private isExpired(task: LoopTask): boolean;

  // 内部：生成 jitter 偏移量
  private calculateJitter(taskId: string, intervalMs: number): number;

  // 内部：生成 8 位任务 ID
  private generateTaskId(): string;
}
```

**执行逻辑关键点：**

- 使用 `setTimeout` 而非 `setInterval`，每次执行后重新安排下一次（避免任务堆积）
- 通过 `executeCallback` 与 AgentSession 集成：
  - agent 空闲时：直接调用 `session.prompt(task.prompt)`
  - agent 忙碌时：调用 `session.followUp(task.prompt)` 入队等待
- 每次执行前检查有效期，过期则自动删除
- jitter 基于 taskId 的哈希值确定性计算，保证同一任务每次偏移一致

### 4.4 AgentSession 集成

> **当前实现**：未在 `AgentSession` 内挂载调度器；扩展中通过 `pi.sendUserMessage(…, { deliverAs: "followUp" })` 与空闲时的无选项调用，等价于下述 `_executeLoopTask` 分支逻辑（消息带 `[Loop Task <id>]` 前缀）。清理在扩展监听 `session_shutdown` 时 `dispose`。

在 `AgentSession` 中添加（**参考/等价逻辑**，非当前代码路径）：

```typescript
// 新增属性
private _loopScheduler: LoopScheduler | null = null;

// 新增方法
get loopScheduler(): LoopScheduler {
  if (!this._loopScheduler) {
    this._loopScheduler = new LoopScheduler(
      {},
      (task) => this._executeLoopTask(task),
    );
  }
  return this._loopScheduler;
}

private async _executeLoopTask(task: LoopTask): Promise<void> {
  const loopPrefix = `[Loop Task ${task.id}] `;
  const promptText = loopPrefix + task.prompt;
  if (this.isStreaming) {
    await this.followUp(promptText);
  } else {
    await this.prompt(promptText, { expandPromptTemplates: true });
  }
}
```

在 `dispose()` 方法中调用 `this._loopScheduler?.dispose()` 清理所有定时器。

## 五、安全与费用控制

### 5.1 费用保护

- **3 天硬性有效期**：防止遗忘的循环任务持续消耗 API 额度
- **最大 50 个任务**：防止任务爆炸
- **最小 1 分钟间隔**：防止高频调用
- **无补录策略**：忙碌期间漏掉的执行不补偿，避免任务堆积

### 5.2 执行安全

- Loop 任务的提示词通过与正常用户输入相同的路径处理（包括扩展拦截、权限检查等）
- 任务执行前加注 `[Loop Task <id>]` 前缀，方便在会话中识别自动化任务
- 支持通过 `/loop delete` 或 `/loop clear` 随时终止

### 5.3 沙箱继承

- Loop 任务在当前 AgentSession 的安全沙箱中执行
- 继承当前会话的所有工具权限和文件系统限制
- 不引入额外的权限提升路径

## 七、与现有架构的适配分析

### 7.1 利用现有 agent-loop 机制

NanoPencil 的 `packages/agent-core/src/agent-loop.ts` 已经实现了完整的 agent 执行循环，包括：
- `getFollowUpMessages` 钩子：agent 完成后检查后续消息
- `getSteeringMessages` 钩子：agent 执行中检查中断消息

`/loop` 任务可以直接利用这两个钩子来注入定时任务的提示词，无需修改核心循环逻辑。

### 7.2 与 AgentSession 的 followUp/steer 对接

`AgentSession` 已提供 `followUp()` 和 `steer()` 方法，`/loop` 调度器直接调用：
- agent 空闲 → `session.prompt()` 直接触发新 turn
- agent 忙碌 → `session.followUp()` 排队等待

### 7.3 InteractiveMode 命令模式

现有斜杠命令通过 `onSubmit` 回调中的 `if/else` 链处理，`/loop` 遵循相同模式，零架构变更。

### 7.4 扩展系统兼容

通过 `session.prompt()` 路径发送的 loop 消息会经过完整的扩展事件链（input hook、skill expansion 等），保持与手动输入的行为一致性。
